### B — SF-047 subdomain auth + rate-limit (`functions/src/subdomainService.ts:172,179`)
- Implementation: `if (!request.auth) throw HttpsError("unauthenticated")` followed by `checkRateLimit('subdomain_check:${uid}', 30, 300)`.
- **Anon test**: POST without `Authorization` header → `401 UNAUTHENTICATED` ✅
- **Initial burst test FAILED METHOD**: `for (...) promises.push(fetch(...))` initiated 34 PARALLEL requests despite never awaiting; burned the rate-limit budget before the sequential burst of 34 more. Tally: 2 success + 33 RESOURCE_EXHAUSTED — falsely suggested limit fires at 2, not 30. Advisor catch — credit to advisor reviewer.
- **Clean re-run with throwaway user** `USVVzpSBQCfh7yzYbf9kJr48lNT2`: 35 strictly-sequential calls → 30 × 200 OK + 5 × 429 RESOURCE_EXHAUSTED, first fail at call #31 — matches code spec exactly. Throwaway user deleted post-test.

### C — SF-048 deleteUserAccount cooldown (`functions/src/deleteUserAccount.ts:62`)
- Implementation: `if (!checkRateLimit('delete_account:${uid}', 1, 300)) throw "resource-exhausted"`. 1 call / 5min window.
- **Structural test design flaw**: sequential 1st call deletes the user → 2nd call's ID token rejected by Firebase Auth with `unauthenticated`, never reaches cooldown logic. Originally skipped per user.
- **Concurrent-call re-test on throwaway user** `LMgycw5ES0aT9YINmZDUG5Khks82`: `Promise.all([fetch, fetch])` raced both past auth check; results — 1 × 200 OK (2469ms full cascade) + 1 × 429 RESOURCE_EXHAUSTED (431ms instant rejection). ✅ Cooldown gate fires on concurrent attempt.
- Pattern learning: for `checkRateLimit(key, 1, N)` gates AFTER auth, concurrent calls are the ONLY way to exercise the limit without invalidating auth first.
