## 2. IP rate-limit design tradeoff (open, by design)

`functions/src/loginLockout.ts:42â€“43` caps `recordLoginFailure` at **1 call per 60s per IP** (hashed). This is a deliberate DoS-mitigation choice but has a UX side effect:

> **A legitimate user who mistypes their password 5Ă— in rapid succession from the same browser will only see the server counter go from 0 â†’ 1** â€” calls 2â€“5 receive `HttpsError("resource-exhausted", "Too many failure reports from this IP.")` and are silently swallowed by `rate_limit_service.dart:159â€“170` (fail-open contract).

The lockout state is reachable in **two real-world scenarios only**:

1. **Slow user, same IP** â€” attempts spaced â‰Ą65s apart. After 5 such attempts the doc reaches `attemptCount = 5` and `lockedUntil = now + 15min`.
2. **Distributed attacker** â€” 5+ different IPs each calling once. Each IP contributes 1 to the counter; lockout fires on the 5th distinct source.

Scenario (2) is the residual DoS surface acknowledged in `functions/src/loginLockout.ts:22â€“25`:

```
Residual risk: a distributed attacker (botnet) can still bump victim's
counter via many IPs. Full closure requires App Check enforcement
(gated on RECAPTCHA_SITE_KEY provisioning + client init per
docs/TODO.md "App Check launch checklist").
```

App Check (`F-50-05` in `docs/TODO.md`) is the gating fix for scenario (2). Audit-only mode shipped via SF-046 already covers `getUnitAvailability` + `createStripeCheckoutSession`; extending it to the 3 lockout callables is the next critical-path item once `RECAPTCHA_SITE_KEY` is provisioned.

### Implication for support / triage

If a user reports "I locked myself out after 5 wrong passwords":
- **Likely**: they typed slowly (â‰Ą65s gaps) â€” wait 15min or have admin reset via `loginAttempts/{docId}` delete.
- **Unlikely**: they typed quickly â€” only 1 attempt would have counted, so the lockout came from a prior session (last 60min) or a different IP source on the same email.
