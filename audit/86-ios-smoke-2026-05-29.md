## Findings — iOS-specific env wiring

These are NOT smoke results — they're code-level observations made during pre-flight that the dev-plist swap does NOT cover. Documenting so future iOS-flavor work can address them in a single Xcode-schemes PR.

### F-86-02 — `Info.plist` GIDClientID hardcoded PROD

`ios/Runner/Info.plist:88-89`:
```xml
<key>GIDClientID</key>
<string>592597958982-a38onq6ft57de2dprnsi3lg0rog27bvg.apps.googleusercontent.com</string>
```

Project `592597958982` is the PROD Firebase project's OAuth client. The dev plist (`GoogleService-Info.plist.backup`, PROJECT_ID `bookbed-dev`) does NOT include CLIENT_ID/REVERSED_CLIENT_ID fields, so `GoogleSignIn` on iOS reads the Info.plist fallback — meaning even with the dev plist active, native Google Sign-In runs the PROD OAuth client, then hands the token to Firebase Auth on `bookbed-dev`. The audiences won't match; Firebase Auth will refuse the credential.

**Impact**: native Google SSO on iOS dev is broken end-to-end (regardless of plist swap). Apple SSO + email/password unaffected — they don't read Info.plist for env wiring.

**Fix path**: Xcode schemes (Debug-dev/Debug-prod) with per-env Info.plist files, OR drop the Info.plist GIDClientID and source from `GoogleService-Info.plist`'s CLIENT_ID (then add CLIENT_ID to the dev plist).

### F-86-03 — `CFBundleURLSchemes` reversed-client redirect hardcoded PROD

`ios/Runner/Info.plist:73`:
```xml
<string>com.googleusercontent.apps.592597958982-a38onq6ft57de2dprnsi3lg0rog27bvg</string>
```

Same project ID class as F-86-02. The Google Sign-In SDK calls back to the app via this URL scheme after OAuth — on dev plist active, the scheme works (Apple matches scheme via Info.plist regardless of which Firebase project is running) but the OAuth dance still completes on the PROD client. Resolved together with F-86-02 by the per-env Info.plist fix.

### F-86-04 — `Runner.entitlements` aps-environment + associated-domains hardcoded PROD

`ios/Runner/Runner.entitlements`:
- `aps-environment` = `production` — for real-device dev pushes you want `development`. Simulator doesn't deliver pushes regardless, so smoke-test impact is zero, but future real-device dev pushes will silently fail token registration.
- `associated-domains`: `bookbed.io`, `www.bookbed.io`, `app.bookbed.io`, `view.bookbed.io` — no `bookbed-{owner,widget,admin}-dev.web.app`. Universal Links from dev hosting URLs won't open the app on iOS.

**Fix path**: same Xcode schemes solution as F-86-02; per-env entitlements file per scheme.
